# v0.6.0 Portal Sync Compatibility -- Deployment Runbook

## Overview

Deploys bidirectional sync between the Phoenix mobile app and the Phoenix Portal.
Three stages must execute in strict order. Each stage has verification steps that
must pass before proceeding to the next.

**Deployment order:** Schema -> Edge Functions -> Mobile App Release

| Stage | Repo | What | Command |
|-------|------|------|---------|
| 1 | phoenix-portal | Database schema migrations | `supabase db push` |
| 2 | phoenix-portal | Edge Functions | `supabase functions deploy` |
| 3 | Project-Phoenix-MP | Mobile app release | Build + publish |

**Why this order matters:**
- Edge Functions reference tables/columns from Stage 1. Deploying Edge Functions
  before schema would cause runtime errors.
- The mobile app calls Edge Functions from Stage 2. Publishing the app before
  Edge Functions are live would cause sync failures for all users.

## Stage 1: Portal Database Schema

**Repo:** `phoenix-portal`
**Migration files (Phase 23):**
- `supabase/migrations/20260228_sync_compat_*.sql` -- schema additions (new tables, columns)
- `supabase/migrations/20260228_mode_data_migration.sql` -- workout mode SCREAMING_SNAKE migration
- `supabase/migrations/20260228_quality_fixes.sql` -- `(select auth.uid())` wrapper, column type fixes

### Deploy

```bash
cd phoenix-portal
supabase link --project-ref <PROJECT_REF>
supabase db push
```

### Verify

1. **Check new tables exist** (Supabase Dashboard -> Table Editor or SQL Editor):

```sql
-- Verify rpg_attributes table
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'rpg_attributes'
ORDER BY ordinal_position;
-- Expected columns: user_id, strength, power, stamina, consistency, mastery,
--   character_class, level, experience_points, created_at, updated_at

-- Verify earned_badges table
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'earned_badges'
ORDER BY ordinal_position;
-- Expected columns: id, user_id, badge_id, badge_name, badge_description,
--   badge_tier, earned_at, created_at, updated_at

-- Verify gamification_stats table
SELECT column_name, data_type
FROM information_schema.columns
WHERE table_name = 'gamification_stats'
ORDER BY ordinal_position;
-- Expected columns: id, user_id, total_workouts, total_reps, total_volume_kg,
--   longest_streak, current_streak, total_time_seconds, created_at, updated_at
```

2. **Check routine_exercises advanced columns:**

```sql
SELECT column_name FROM information_schema.columns
WHERE table_name = 'routine_exercises'
  AND column_name IN (
    'superset_id', 'rest_seconds', 'tempo', 'notes',
    'drop_set_enabled', 'drop_set_percentage', 'drop_set_sets',
    'echo_enabled', 'echo_percentage', 'echo_reps', 'echo_sets', 'echo_level'
  );
-- Should return 12 rows
```

3. **Check RLS policies on INSERT:**

```sql
SELECT tablename, policyname, cmd
FROM pg_policies
WHERE schemaname = 'public'
  AND cmd = 'INSERT'
  AND tablename IN ('exercises', 'sets', 'rep_summaries', 'rep_telemetry');
```

4. **Verify mode migration:**

```sql
SELECT DISTINCT workout_mode FROM workout_sessions LIMIT 10;
-- Should show SCREAMING_SNAKE values: CONCENTRIC, ECCENTRIC, OLD_SCHOOL, etc.
-- Should NOT show: Concentric, concentric, Classic, Power, etc.
```

5. **Verify auth.uid() wrapper (quality fix):**

```sql
-- Check that RLS policies use (select auth.uid()) not bare auth.uid()
SELECT policyname, qual
FROM pg_policies
WHERE schemaname = 'public'
  AND qual LIKE '%auth.uid()%'
  AND qual NOT LIKE '%(select auth.uid())%';
-- Should return 0 rows (all policies use the select wrapper)
```

### Rollback

Schema changes are additive (new tables, new columns, new policies). They do not
modify existing data structures, so rollback is rarely needed.

If a migration fails partway through:
```bash
# Fix the failing SQL file, then re-run
supabase db push
# Migrations are tracked -- already-applied migrations will be skipped
```

If you must fully revert (destructive -- data loss):
```sql
-- Only if absolutely necessary
DROP TABLE IF EXISTS rpg_attributes CASCADE;
DROP TABLE IF EXISTS earned_badges CASCADE;
DROP TABLE IF EXISTS gamification_stats CASCADE;
-- Column drops from routine_exercises would also be needed
```

## Stage 2: Edge Functions

**Repo:** `phoenix-portal`
**Functions (Phase 25):**
- `supabase/functions/mobile-sync-push/` -- receives workout data from mobile
- `supabase/functions/mobile-sync-pull/` -- sends portal data to mobile

### Deploy

```bash
cd phoenix-portal
supabase functions deploy mobile-sync-push
supabase functions deploy mobile-sync-pull
```

### Verify

First, obtain a valid GoTrue access token for testing:

```bash
# Get a test token (replace with real credentials)
curl -X POST 'https://<PROJECT_REF>.supabase.co/auth/v1/token?grant_type=password' \
  -H "apikey: <ANON_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"testpassword"}'
# Copy the access_token from the response
```

**Test push endpoint:**

```bash
curl -X POST 'https://<PROJECT_REF>.supabase.co/functions/v1/mobile-sync-push' \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "apikey: <ANON_KEY>" \
  -H "Content-Type: application/json" \
  -d '{
    "deviceId": "deploy-test",
    "platform": "android",
    "lastSync": 0,
    "sessions": [],
    "routines": [],
    "badges": [],
    "rpgAttributes": null,
    "gamificationStats": null
  }'
```
**Expected:** HTTP 200 with JSON containing `syncTime`, `sessionsInserted: 0`, etc.

**Test pull endpoint:**

```bash
curl -X POST 'https://<PROJECT_REF>.supabase.co/functions/v1/mobile-sync-pull' \
  -H "Authorization: Bearer <ACCESS_TOKEN>" \
  -H "apikey: <ANON_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"deviceId": "deploy-test", "lastSync": 0}'
```
**Expected:** HTTP 200 with JSON containing `syncTime`, `sessions: []`, `routines: []`, etc.

**Test without Origin header (mobile client simulation):**
The above `curl` commands have no `Origin` header by default, simulating a mobile
client. Both should return 200. If you see a CORS error, the Edge Function has an
Origin check that needs to be relaxed for mobile clients.

**Test unauthorized access:**

```bash
curl -X POST 'https://<PROJECT_REF>.supabase.co/functions/v1/mobile-sync-push' \
  -H "apikey: <ANON_KEY>" \
  -H "Content-Type: application/json" \
  -d '{"deviceId": "test"}'
```
**Expected:** HTTP 401 (no Bearer token = unauthorized)

### Rollback

```bash
# Deploy previous version of functions
# Check deployment history in Supabase Dashboard -> Edge Functions
supabase functions deploy mobile-sync-push --version <PREVIOUS_VERSION>
supabase functions deploy mobile-sync-pull --version <PREVIOUS_VERSION>
```

If the functions need to be taken offline entirely:
```bash
supabase functions delete mobile-sync-push
supabase functions delete mobile-sync-pull
```
Note: Deleting functions will cause the mobile app to show sync errors (non-fatal --
the app continues to work offline).

## Stage 3: Mobile App Release

**Repo:** `Project-Phoenix-MP`

### Pre-release Checks

- [ ] Edge Functions are deployed and verified (Stage 2 complete)
- [ ] Build succeeds: `./gradlew :androidApp:assembleRelease`
- [ ] Tests pass: `./gradlew :androidApp:testDebugUnitTest`
- [ ] Version code and version name bumped in `androidApp/build.gradle.kts`
- [ ] Changelog updated with v0.6.0 changes:
  - Bidirectional sync with Phoenix Portal
  - Supabase GoTrue authentication (email/password)
  - Workout, routine, gamification, and badge syncing
- [ ] Supabase project URL and anon key configured in `local.properties` / BuildConfig
- [ ] No hardcoded service_role keys in mobile code (security check)

### Deploy -- Android

```bash
cd Project-Phoenix-MP
./gradlew :androidApp:bundleRelease
# Output: androidApp/build/outputs/bundle/release/androidApp-release.aab
```

1. Upload AAB to Google Play Console
2. Create release in Internal Testing track first
3. Test on physical device (see Verify below)
4. Promote to Closed Beta -> Open Beta -> Production (staged rollout recommended)

### Deploy -- iOS

1. Open `iosApp/` in Xcode
2. Build shared framework: `./gradlew :shared:assembleXCFramework`
3. Archive in Xcode (Product -> Archive)
4. Upload to App Store Connect
5. Submit for TestFlight review, then App Store review

### Verify

Perform these checks on a physical device with the release build:

1. **Fresh login:**
   - Open app -> navigate to sync/portal settings
   - Sign in with portal email/password
   - Verify: login succeeds, user info displayed

2. **Push sync (mobile -> portal):**
   - Complete a workout on the trainer
   - Trigger sync (automatic or manual)
   - Check Supabase Dashboard -> Table Editor -> `workout_sessions`
   - Verify: new session row appears with correct mode (SCREAMING_SNAKE), weight, reps

3. **Pull sync (portal -> mobile):**
   - Create or modify a routine in the portal web UI
   - Trigger sync on mobile
   - Verify: routine appears/updates on mobile

4. **Session persistence:**
   - Force-kill the app
   - Reopen the app
   - Verify: still logged in (no login prompt), last sync time preserved

5. **Offline resilience:**
   - Enable airplane mode
   - Complete a workout
   - Disable airplane mode, trigger sync
   - Verify: queued workout syncs successfully

### Rollback

**Android:**
- Google Play Console -> halt staged rollout
- If critical: unpublish the release and re-publish the previous version

**iOS:**
- App Store Connect -> remove from sale
- Submit previous version as expedited review

**Important:** Edge Functions remain deployed and backwards-compatible. Rolling back
the mobile app does not require rolling back Edge Functions.

## Post-Deployment Monitoring

### First 24 Hours

- [ ] Monitor Supabase Edge Function logs (Dashboard -> Edge Functions -> Logs)
  - Watch for: 500 errors, timeout errors, auth failures
- [ ] Monitor mobile crash reports (Firebase Crashlytics / Sentry)
  - Watch for: `PortalApiException`, network timeouts, JSON parse errors
- [ ] Check sync success rate:

```sql
-- Sessions synced in last 24 hours
SELECT COUNT(*), DATE_TRUNC('hour', created_at) as hour
FROM workout_sessions
WHERE created_at > NOW() - INTERVAL '24 hours'
GROUP BY hour
ORDER BY hour;
```

### First Week

- [ ] Review Edge Function invocation counts and error rates
- [ ] Check for token refresh failures (users getting logged out unexpectedly)
- [ ] Verify no duplicate sessions (same client_id appearing multiple times):

```sql
SELECT client_id, COUNT(*)
FROM workout_sessions
GROUP BY client_id
HAVING COUNT(*) > 1;
-- Should return 0 rows
```

- [ ] Monitor database size growth (Dashboard -> Database -> Reports)

### Ongoing

- [ ] Set up Supabase alerts for Edge Function error rate > 5%
- [ ] Set up uptime monitoring for Edge Function endpoints
- [ ] Review sync latency metrics monthly
