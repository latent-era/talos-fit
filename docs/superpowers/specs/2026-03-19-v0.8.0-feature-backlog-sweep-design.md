## Hardware Constraints

1. **No mid-set BLE interruptions.** Pausing or interrupting BLE communication during an active set faults the Vitruvian machine hardware. All "pause" or "stop" features must operate between sets only. The sole exception is the voice emergency stop (#141), which intentionally triggers the existing stop action.
2. **No overlapping exercise packets.** The machine cannot receive a new exercise packet until the active one is fully ended. Sending one mid-exercise causes a machine fault. Variable warm-up sets (#30) require research into the BLE packet lifecycle before implementation.

## Sequencing Strategy

Approach B: complete what's started (Smart Just Lift has Phase 11 foundation), then sweep quick wins outward by tier.

## Phase 32: Smart Just Lift + Motion Start

**Issues:** #233, #237
**Tier:** T4
**Estimated effort:** 2 weeks

### 32A — Post-set exercise label picker (#233)

Wire the existing exercise auto-detection infrastructure (Phase 11: `SignatureExtractor`, `ExerciseClassifier`, `ExerciseDetectionManager`) into Just Lift mode's set-completion flow.

**Detection timing:** The existing `ExerciseDetectionManager.onRepCompleted()` triggers after `MIN_REPS_FOR_DETECTION = 3` reps (mid-set). For Just Lift, detection runs mid-set but the picker UI is deferred to post-set — the classifier accumulates data during the set and presents its result when the set ends. This gives the classifier more data points while keeping the picker non-intrusive (user is resting, attention is free).

**Behavior:**
- During a Just Lift set, `ExerciseDetectionManager` accumulates rep data and runs classification after 3+ reps (existing behavior)
- After the set completes, present the `AutoDetectionSheet` bottom sheet showing the classifier's best guess plus 2–3 alternatives ranked by similarity score
- User taps to confirm or picks a different exercise
- Selected exercise is associated with that set's data via the existing `exerciseId` and `exerciseName` columns on `WorkoutSession` (already nullable in the schema — Just Lift currently writes null; this populates them)
- **Auto-accept (new behavior):** If classifier confidence > 0.90, auto-accept and show a brief toast instead of the full picker — reduces friction for exercises the system knows well. Note: the existing `HISTORY_MATCH_THRESHOLD = 0.85` controls classifier fallback selection, not user-facing auto-accept. The auto-accept threshold is intentionally higher (0.90) since it skips user confirmation entirely
- First-time users with no signature history get a full exercise list picker (manual fallback)

**Reuse:** The `AutoDetectionSheet` composable from Phase 11 already exists and is rendered from `WorkoutHud.kt` (not `JustLiftScreen.kt`). `JustLiftScreen` is the setup screen that clears `selectedExerciseId` and navigates into the active workout flow. The actual post-set detection wiring targets `WorkoutHud` / `ActiveWorkoutScreen` — the same layer where detection already operates for routine workouts.

**Known gap:** `AutoDetectionSheet` alternate exercise chips currently pass empty string IDs via `onConfirm("", alternateName)`. This must be fixed: add an exercise-by-name lookup (via `ExerciseRepository`) so alternates resolve to valid exercise IDs before calling `ExerciseDetectionManager.onExerciseConfirmed()`.

**Schema:** No migration needed — `WorkoutSession` already has nullable `exerciseId TEXT` and `exerciseName TEXT` columns. Phase 32 populates these for Just Lift sets.

**Key files:**
- `WorkoutHud.kt` / `ActiveWorkoutScreen.kt` — wire `AutoDetectionSheet` into Just Lift post-set flow (detection sheet already rendered here)
- `ExerciseDetectionManager.kt` — add Just Lift-specific entry point, defer picker to post-set
- `AutoDetectionSheet.kt` — fix alternate chip exercise ID resolution
- `MainViewModel.kt` — expose exercise attribution for Just Lift sessions

### 32B — Motion-triggered set start for Routines & Single Exercise (#237)

Port the existing Just Lift auto-start logic (cable-hold detection) to the Single Exercise and Routine workout flows as an opt-in replacement for the 5-second "get ready" countdown.

**Behavior:**
- Extract the auto-start detection logic from `WorkoutCoordinator` (where the auto-start state machine lives — `autoStartCountdown`, handle detection) into a reusable `MotionStartDetector` utility that watches the `MetricSample` stream for sustained cable load
- Wire `MotionStartDetector` into `ActiveWorkoutScreen` — when enabled, the "Get Ready" countdown is replaced with the cable-hold detection + visual indicator
- User setting: "Motion-triggered set start" (default: off for routines/single exercise, preserving existing behavior)
- The existing 5-second countdown remains the default and fallback

**Key files:**
- `WorkoutCoordinator.kt` — extract auto-start logic into shared utility
- New: `MotionStartDetector.kt` — reusable cable-hold detection from MetricSample stream
- `ActiveWorkoutScreen.kt` — wire motion start as countdown alternative
- `ActiveSessionEngine.kt` — support motion-start state alongside countdown state
- `SettingsTab.kt` / `PreferencesManager.kt` — motion start preference

## Phase 33: Workout Flow QoL

**Issues:** #201, #297, #228, #113
**Tier:** T0–T3
**Estimated effort:** 1 week

### 33A — Show combined cable weight in Just Lift setup (#201)

The active workout HUD (`WorkoutHud.kt` line 1048) already displays `totalLoad` during an active set. The gap is the **Just Lift setup screen** — where the user configures weight before starting, only individual per-cable weights are shown. Add a "Total" display below the two individual cable weight readouts on the setup portion of `JustLiftScreen`. Pure calculation: `total = cableA + cableB`. Respects weight unit preference (kg/lb) and uses existing `formatWeight` utility.

**Key files:** `JustLiftScreen.kt` only — one `Text` composable beneath existing cable weight display on the setup screen. Scope is narrow since active-workout total display already exists.

### 33B — +30 Seconds button on rest countdown (#297)

Add a tappable "+30s" button beside the rest timer circle on the between-set rest screen. Each tap adds 30 seconds cumulatively (no cap). Extended time is reflected in the circular countdown animation.

**Implementation note:** The existing `RestTimerCard` composable receives `restSecondsRemaining` as a parameter from `ActiveSessionEngine` — it has no internal countdown state. The actual countdown runs in `ActiveSessionEngine`. Therefore, +30s must modify the engine's rest timer state (add to remaining duration), not just the composable.

**Key files:**
- `ActiveSessionEngine.kt` — `extendRestTime(seconds: Int)` method on the rest timer state machine
- `RestTimerCard.kt` — add "+30s" button that calls the engine method

### 33C — Pause/restart rest timer (#228)

Add pause/play and restart controls to the rest timer:
- **Pause/resume:** Tapping pauses the countdown; tapping again resumes from where it left off.
- **Restart:** A separate reset button restarts the timer to its original duration (the configured rest time, plus any +30s extensions applied before the reset). This covers the "restart" aspect of #228.

Combined with 33B, this gives full rest timer control: extend (+30s), pause, resume, restart.

**Implementation:** `isPaused` and `originalDuration` state in `ActiveSessionEngine`'s rest timer coroutine. When paused, the countdown coroutine suspends. `resetRestTimer()` sets remaining time back to `originalDuration`. `RestTimerCard` renders pause/play and restart buttons.

**Key files:**
- `ActiveSessionEngine.kt` — `toggleRestPause()`, `resetRestTimer()` methods, `isPaused`/`originalDuration` state
- `RestTimerCard.kt` — pause/play toggle + restart button

### 33D — Rest timer for Just Lift mode (#113)

Just Lift currently has no inter-set rest timer. After a set completes, show a configurable rest countdown.

**Interaction with Phase 32A:** When the exercise label picker (32A) is enabled, the sequencing is: set ends → exercise detection sheet appears → user confirms/auto-accept → rest timer begins. The rest timer starts **after** detection confirmation, not concurrently. This keeps the UX sequential and predictable.

**Behavior:**
- Rest duration set via scrollable dial on Just Lift setup screen, default 60 seconds
- Reuses the same rest timer component from 33B/33C, including +30s and pause controls

**Key files:**
- `JustLiftScreen.kt` — rest duration picker on setup, rest timer between sets
- `ActiveSessionEngine.kt` — Just Lift rest timer state (reusing the same rest timer state machine as routines)

**Design note:** Items B, C, and D all touch the rest timer. Building B+C first enhances `ActiveSessionEngine`'s rest timer state machine with extend/pause, then D reuses it in Just Lift. The Just Lift rest timer gets +30s and pause/resume for free.

## Phase 34: Workout Navigation

**Issue:** #152
**Tier:** T2
**Estimated effort:** 1 week

**Existing foundation:** `ExerciseNavigator` composable (with `onPrevious`/`onNext`/`canGoBack`/`canSkipForward`) and `RoutineFlowManager.jumpToExercise(index)` already exist, including backward navigation (`prevIndex` at line 907, `canGoBack()` at line 913) wired through `MainViewModel.jumpToExercise()` into `ActiveWorkoutScreen`. This phase is **hardening and completing** the existing navigation, not greenfield work. The remaining gaps are: skip tracking/persistence for session history accuracy, and ensuring navigation controls are correctly gated to between-set timing only.

**Scope constraints:**
- Only applies to Routine workouts (multiple exercises in sequence). Single Exercise and Just Lift have no "next/previous" concept.
- Navigation happens between sets only (per machine fault constraint). Controls appear on the rest timer screen and exercise intro screen.
- **Supersets:** Navigation operates at the exercise-group level. Forward/back skips entire superset groups, not individual exercises within a superset. Skipping mid-superset marks the remaining exercises in that group as skipped. This avoids disrupting `RoutineFlowManager`'s superset cycling logic.

**Behavior:**
- Left/right arrow buttons: `< Previous Exercise` / `Next Exercise >`
- **Forward skip:** Marks the current exercise as "skipped" (records 0 completed sets for traceability), advances to next. Disabled on last exercise.
- **Backward navigation:** Returns to previous exercise. User can add more sets to an already-completed exercise. Previously recorded sets are preserved — navigation appends, never overwrites.
- Existing position indicator ("Exercise 3 of 8") becomes interactive.

**State management:**
- `ActiveSessionEngine` already tracks `currentExerciseIndex`. Navigation updates this index and resets the per-exercise set counter.
- `skippedExercises: Set<Int>` tracks which exercises were skipped for session history accuracy.
- BLE: no changes. Next exercise's weight/mode is sent when the new exercise begins, same as normal progression.

**Key files:**
- `RoutineFlowManager.kt` — `jumpToExercise(index)` already exists; add skip tracking (`skippedExercises: Set<Int>`)
- `ExerciseNavigator.kt` — already exists; enhance with skip/back affordances gated to between-set state
- `ActiveWorkoutScreen.kt` — already wires `onJumpToExercise`; ensure controls only appear between sets
- `WorkoutSession` logging — record skipped exercises (may need a column or JSON field for persistence)

## Phase 35: Routine Builder Enhancements

**Issues:** #29, #259, #30
**Tier:** T2–T3
**Estimated effort:** 2 weeks

All three features modify the routine editor and execution. They share a dependency on the routine data model. No new schema migrations needed — 35A and 35C use columns already added by migration 18 (`setEchoLevels TEXT`, `warmupSets TEXT`), and 35B is a behavioral change using existing `setRestSeconds`/`perSetRestTime` fields.

### 35A — Per-set Echo level in routines (#29)

Allow each individual set in a routine to specify its own Echo level, enabling progressive patterns like "Hard, Harder, Hardest, Epic" within a single exercise.

**Echo levels:** Hard, Harder, Hardest, Epic (4-value `EchoLevel` enum, not numeric).

**Data model:** The DB column `setEchoLevels TEXT` already exists (added by migration 18). The work here is Kotlin-side: add `setEchoLevels: List<EchoLevel?>` to the `RoutineExercise` domain model in `Routine.kt`, add repository mapping to deserialize the TEXT column (JSON array of enum names, matching the `setRestSeconds` serialization pattern), and wire it through the routine editor and execution engine. No schema migration needed.

**Behavior:**
- Routine editor: when exercise mode is Echo, each set row gets a compact 4-option selector (segmented button or dropdown). Defaults to exercise-level value.
- Execution: `ActiveSessionEngine` reads `setEchoLevels[setIndex]` (falling back to exercise-level `echoLevel` when null) and sends appropriate BLE packet before each set.

**Key files:** `Routine.kt` (`RoutineExercise` — add field + serialization), repository mapping layer, routine editor composables, `ActiveSessionEngine.kt`.

### 35B — Per-exercise rest times within supersets (#259)

Allow each exercise within a superset to use its own rest period instead of sharing one value.

**Existing state:** `RoutineExercise` already has `setRestSeconds: List<Int>`, `perSetRestTime: Boolean`, and `getRestForSet(setIndex)` for per-set rest within a single exercise. `Superset` has `restBetweenSeconds: Int` shared across all exercises. Currently, superset execution always uses `Superset.restBetweenSeconds`, ignoring each exercise's own rest settings.

**Behavior — no new column needed:**
- The fix is behavioral, not schema-based: when executing a superset, check each exercise's `perSetRestTime` flag. If `true`, use `exercise.getRestForSet(setIndex)` instead of `superset.restBetweenSeconds`. If `false`, fall back to the superset-level rest (current behavior, backward compatible).
- Routine editor: when editing a superset, each exercise row shows a toggle to "Use own rest times" (maps to existing `perSetRestTime`). When enabled, the exercise's per-set rest values apply within the superset context. A "Same for all" toggle disables per-exercise rest and uses the superset default.
- This reuses the existing `setRestSeconds` data and `perSetRestTime` toggle — no migration, no new column, no new DTO field.

**Key files:**
- `ActiveSessionEngine.kt` — change superset rest resolution to check `exercise.perSetRestTime`
- Routine editor superset UI — expose `perSetRestTime` toggle per exercise within superset view

### 35C — Variable warm-up sets (#30)

Let users define custom warm-up sets before the first working set of each exercise.

> **RESEARCH GATE:** Before implementation, investigate the BLE exercise packet lifecycle. The machine cannot receive a new exercise packet until the active one is fully ended. Variable warm-up sets change the weight between sets within a single exercise, which may conflict with this constraint. Research must determine: how does the current routine execution transition between exercises? Can that mechanism be reused for intra-exercise weight changes? If not feasible without firmware changes, descope #30 or redesign warm-up sets as separate "exercises" in the routine sequence.

**Behavior (pending research gate):**
- Add `warmupSets: List<WarmupSet>` to the `RoutineExercise` domain model, where `WarmupSet` has `reps: Int` and `percentOfWorking: Int` (e.g., 50%, 70%, 90%).
- Routine editor: collapsible "Warm-up" section per exercise with "Add warm-up set" button. Common presets: "Standard (50/70/90%)", "Light (60/80%)", "None".
- Execution: warm-up sets execute first with weight computed as `workingWeight * percent / 100`. Visually distinguished in HUD ("Warm-up 2/3" label, muted color). Do NOT count toward working set totals or PR tracking.

**Schema:** The DB column `warmupSets TEXT` already exists (added by migration 18) as a denormalized TEXT field. Use JSON-serialized warm-up data in this existing column (e.g., `[{"reps":5,"percent":50},{"reps":3,"percent":70}]`), matching the existing serialization pattern used by `setRestSeconds`. No new table or migration needed — the work is Kotlin-side domain model + serialization.

**Key files:** `Routine.kt` (add `WarmupSet` data class + `warmupSets` field with JSON serialization), repository mapping layer, routine editor UI, `ActiveSessionEngine.kt`.

## Phase 36: Data Portability

**Issue:** #293
**Tier:** T3
**Estimated effort:** 3–4 days

Automatically write a JSON backup file to a user-accessible location on the device filesystem after each workout.

**Behavior:**
- Settings toggle: "Auto-backup workouts" (default: off)
- After each `WorkoutSession` is saved to SQLite, write a JSON backup to:
  - **Android:** App-specific external files directory (`getExternalFilesDir("PhoenixBackups")`) — no permissions required, accessible via file manager, but removed on app uninstall
  - **iOS:** App documents directory with `UIFileSharingEnabled` — visible in Files app under Phoenix
- Filename: `phoenix-workout-{ISO-date}-{sessionId}.json`
- No network involved — purely local filesystem writes
- "Backup History" section in Settings shows count and total size of backups, with "Open backup folder" shortcut

**Backup format — new per-session export:**
The existing `DataBackupManager` only supports full-dataset export (`exportAllData()`, `exportToJson()`, `exportToFile()`). Auto-backup requires a **new** `exportSession(sessionId)` method that serializes a single `WorkoutSession` with its nested exercises, sets, and rep data — not the entire database. This is a new capability on `DataBackupManager`, not a reuse of existing methods. The JSON structure should follow the same `BackupData` schema but scoped to one session (a `BackupData` with a single-element sessions list), ensuring restore compatibility with the existing `importBackup()` path.

**Key files:**
- `DataBackupManager.kt` — add `exportSession(sessionId)` method (new per-session export)
- `ActiveSessionEngine.kt` — call auto-backup after session save
- `SettingsTab.kt` / `SettingsManager.kt` — toggle + backup info display
- `expect/actual` for platform filesystem paths (Android `getExternalFilesDir`, iOS documents directory)

## Phase 37: Multi-Language Support

**Issue:** #238
**Tier:** T4
**Estimated effort:** 2–3 weeks

Extract all user-visible strings into resource files and ship translated bundles.

**Scope note:** Issue #238 originally requested Dutch translation. During design, the maintainer expanded scope to 5 languages based on install base analysis (US, UK, Australia, Austria, Switzerland, Belgium, Mexico) and approved the community review infrastructure. This expansion is intentional, not scope creep.

### Resource Strategy

Use Compose Multiplatform's built-in `org.jetbrains.compose.resources` string resources (stable in Compose MP 1.10.1). Strings defined in `shared/src/commonMain/composeResources/values/strings.xml` (default English) with locale variants in `values-nl/`, `values-de/`, etc. Accessed via generated `Res.string.xxx` references.

### Target Languages

| Language | Code | Justification |
|----------|------|---------------|
| English | `en` | Default — 75%+ of install base |
| Dutch | `nl` | Specifically requested by a user, Belgium presence |
| German | `de` | Austria + Switzerland — largest non-English segment |
| Spanish | `es` | Mexico + large global fitness app language |
| French | `fr` | Switzerland + Belgium overlap, large global reach |

### Translation Approach

Hybrid: AI-generate + community review.

1. **Initial translations:** AI-generated with translator notes for fitness terminology context (e.g., "set" = group of repetitions, "rep" = single repetition). Some terms may stay in English across languages (e.g., "1RM" is universal).
2. **Community review:** Crowdin (free for open-source) or Weblate for community members to review, correct, and vote on translations. Syncs with Git repo, exports to `strings.xml` format.
3. **In-app:** "Help improve translations" link in Settings under language selection, pointing to the review platform.
4. **Beta flag:** Translations marked as "Beta" in the language picker until a native speaker has reviewed them.

### Extraction Scope

- All composables in `presentation/screen/` and `presentation/components/` — every hardcoded string in `Text()`, `placeholder`, `contentDescription`, button labels
- Domain strings: workout mode names, muscle group labels, badge names
- Format strings for dynamic content (e.g., `"Set %1$d of %2$d"`)
- Constants in `Constants.kt` that are user-visible move to resources

### Exclusions

- BLE protocol strings, UUIDs — not user-visible
- Database column names, sync DTO field names — wire format
- Log messages — stay in English

**Key files:**
- New: `shared/src/commonMain/composeResources/values/strings.xml` + `values-nl/`, `values-de/`, `values-es/`, `values-fr/`
- Every file in `presentation/screen/` and `presentation/components/` — string literal → `stringResource()` replacement
- `build.gradle.kts` — compose resources plugin configuration (prerequisite: `composeResources/values/` directory and plugin config may not exist yet — setup required before string extraction begins)

## Phase 38: Voice Emergency Stop

**Issue:** #141
**Tier:** T5
**Estimated effort:** 3–4 weeks

On-device speech recognition detects a user-configured safe word during active workouts and programmatically triggers the existing Stop button action.

### Speech Recognition

- **Android:** `SpeechRecognizer` API with `EXTRA_PREFER_OFFLINE = true` — on-device, no network dependency
- **iOS:** `SFSpeechRecognizer` with `requiresOnDeviceRecognition = true` (effective iOS 15+ for reliable on-device English models; iOS 13-14 may fall back to server-based recognition)
- **expect/actual:** `expect class SafeWordListener` in commonMain with platform implementations

### Detection → Action

Safe word detected → call `viewModel.stopAndReturnToSetReady()` directly, **bypassing the confirmation dialog**. This stops the active set and returns to the set-ready state, allowing the user to resume their workout if the trigger was a false positive.

**Why `stopAndReturnToSetReady()` instead of `stopWorkout(exitingWorkout = true)`:** The `stopWorkout(exitingWorkout = true)` path saves the session and exits to Idle — this is a destructive, non-resumable action. A false positive with that path would end the user's entire workout, not just pause it. `stopAndReturnToSetReady()` stops the current set and returns to set-ready, which is non-destructive and resumable. The machine stops, the user is safe, and they can continue if it was a false trigger.

The confirmation dialog (`showExitConfirmation` → `AlertDialog`) is bypassed because the 3-repetition calibration step already serves as the user's advance consent.

### Safe Word Configuration

- User types their chosen word/phrase in a text field
- **3-repetition calibration:** The app listens and asks the user to say the word 3 times. Each successful detection gets a checkmark. All 3 must pass before the word is saved.
- Calibration validates that the on-device recognizer can reliably detect that word in that user's voice, accent, and environment
- If calibration fails (word not detected reliably), suggest trying a different word
- No syllable minimum — calibration is the primary gate against false positives. If a short word like "STOP" (per issue #141's example) passes 3/3 calibration, it's accepted. The calibration step should ideally be performed in a gym-like environment to test real-world noise conditions.
- Recalibrate anytime from Settings

### Lifecycle

- Continuous background listening during active workout sessions only
- Microphone released when workout ends or app backgrounds
- If recognition engine stops, auto-restart within 1 second
- Coexists with music playback (Android `AUDIOFOCUS_GAIN_TRANSIENT`, iOS `.mixWithOthers`)

### Settings

- Toggle: on/off (default: off — opt-in, requires microphone permission)
- Safe word field + calibration flow
- No tier gating — safety feature for all users

### Failure Handling

- Microphone permission denied → feature disabled with explanation
- Recognition engine unavailable → feature disabled for session, toast notification, workout proceeds
- False positive → `stopAndReturnToSetReady()` is non-destructive; user can immediately start the next set. Current set's partial data is preserved.
- Audio session interrupted (phone call) → listener pauses, auto-resumes when available

**Key files:**
- New: `expect SafeWordListener` + Android/iOS actuals
- New: `SafeWordDetectionManager.kt` — lifecycle, restart, detection callback
- `ActiveWorkoutScreen.kt` — wire listener, call existing stop handler on detection
- `SettingsTab.kt` / `PreferencesManager.kt` — safe word configuration + calibration UI

## Sync DTO Impact

Phase 35 adds new routine data fields (per-set echo levels, per-exercise superset rest override, warm-up sets). These must be reflected in sync DTOs and Edge Functions:
- **Push DTO** (`PortalRoutineExerciseSyncDto`): add `perSetEchoLevels: String? = null` (JSON array string, matching existing `perSetWeights: String?` and `perSetRest: String?` pattern), `warmupSets: String? = null` (JSON array string of `{"reps":N,"percent":N}` objects)
- **Pull DTO** (`PullRoutineExerciseDto`): add the same two fields for round-trip sync
- Phase 35B (superset rest) requires no DTO changes — it reuses existing `perSetRest` semantics
- **Mapping layers** (in addition to DTOs): `PortalSyncAdapter.kt` routine mapping (~line 392) and `SqlDelightSyncRepository.kt` routine sync (~line 446) must be updated to serialize/deserialize the new fields
- Edge Functions `mobile-sync-push` and `mobile-sync-pull`: update routine handling to pass through new fields
- Phase 32A requires no sync changes — `exerciseId` and `exerciseName` are already part of the session sync DTO

This should be handled as a final plan within each affected phase.
