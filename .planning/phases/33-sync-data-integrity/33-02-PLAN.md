## H5: First sync sends entire history with no chunking
- **File:** `shared/src/commonMain/kotlin/com/devil/phoenixproject/data/sync/SyncManager.kt`
- `lastSync = tokenStorage.getLastSyncTimestamp()` returns 0L on first sync
- `selectSessionsModifiedSince(0L)` returns ALL sessions (updatedAt > 0 OR IS NULL)
- No batching — entire history sent in single HTTP request
- If payload exceeds Edge Function body size limit (~1MB), push throws error
- `lastSyncTimestamp` only updated on full success — retry sends same giant payload forever
- User with ~6 months history (360+ sessions with metrics) will hit this

## H6: Push function routine_exercises has no transaction (CROSS-REPO)
- **File:** `C:/Users/dasbl/AndroidStudioProjects/Phoenix App Monorepo/phoenix-portal/supabase/functions/mobile-sync-push/index.ts`
- Lines 756-797: routine_exercises pattern is:
  1. `routines` upsert (safe)
  2. `routine_exercises` DELETE all for these routine_ids (DESTRUCTIVE)
  3. `routine_exercises` INSERT new exercises
- If step 3 fails after step 2 succeeds, routines are empty in Supabase
- Same pattern for cycle_days (lines 802-847)
- No database transaction wraps these operations
- Each `.from().upsert()` is an independent HTTP call to Supabase REST API

## Architecture constraint
- Supabase Edge Functions can't directly run SQL transactions via the REST API
- Options: (a) Supabase RPC function with transaction, (b) upsert-on-conflict instead of delete+insert, (c) soft-delete approach
- Option (b) is simplest: use upsert with ON CONFLICT on (routine_id, exercise_order) instead of delete+reinsert
</context>

## Task 2: Fix routine_exercises delete+insert atomicity (H6)

**Action:** In `phoenix-portal/supabase/functions/mobile-sync-push/index.ts`, replace the destructive delete+insert pattern with upsert-on-conflict for routine_exercises and cycle_days.

**Specific changes:**
1. For routine_exercises (~lines 756-797):
   - REMOVE the `routine_exercises.delete().in('routine_id', routineIds)` call
   - Instead, mark all existing routine_exercises for these routines as stale:
     ```typescript
     // Soft approach: upsert new exercises, then delete orphans
     const exerciseRows = routines.flatMap(r => r.exercises.map((e, i) => ({
       routine_id: r.serverId || r.clientId,
       exercise_order: i,
       exercise_name: e.exerciseName,
       // ... other fields
     })));
     await supabase.from('routine_exercises').upsert(exerciseRows, {
       onConflict: 'routine_id,exercise_order'
     });
     // Delete orphans (exercises beyond the new count)
     for (const routine of routines) {
       await supabase.from('routine_exercises')
         .delete()
         .eq('routine_id', routine.serverId || routine.clientId)
         .gte('exercise_order', routine.exercises.length);
     }
     ```
   - This ensures existing exercises are updated, not deleted+reinserted
   - If the upsert fails, existing data is preserved (no destructive delete happened)

2. Apply the same pattern to cycle_days (~lines 802-847)

3. **Alternative (if upsert-on-conflict isn't feasible due to missing unique constraint):**
   Wrap in a Supabase RPC:
   ```sql
   CREATE OR REPLACE FUNCTION sync_routine_exercises(...)
   RETURNS void AS $$
   BEGIN
     DELETE FROM routine_exercises WHERE routine_id = ANY($1);
     INSERT INTO routine_exercises (...) VALUES ...;
   END;
   $$ LANGUAGE plpgsql;
   ```
   Then call via `supabase.rpc('sync_routine_exercises', {...})`

**Verify:** Code review confirms no unguarded delete+insert pattern remains.

**Done when:** Routine exercises and cycle days cannot be left empty by a partial sync failure.
