## B5: START_STICKY + null intent crashes foreground service
- **File:** `androidApp/src/main/kotlin/com/devil/phoenixproject/service/WorkoutForegroundService.kt`
- Lines 58-74: `onStartCommand` returns `START_STICKY` (line 73)
- `when(intent?.action)` block: neither branch executes on null intent
- Android restarts service with null intent → `startForeground()` never called → `RemoteServiceException` on Android 8+
- **Fix options:**
  A. Handle null intent explicitly — call `startForeground()` then `stopSelf()` (safest)
  B. Switch to `START_NOT_STICKY` — service doesn't restart after kill (simplest, since killed workout can't meaningfully resume)

## B6: Auth tokens in plaintext SharedPreferences

- **File:** `shared/src/androidMain/kotlin/com/devil/phoenixproject/di/PlatformModule.android.kt` (lines 24-27)
- Supabase JWT, refresh token, and email stored in unencrypted `vitruvian_preferences.xml`
- `Settings` backed by `SharedPreferencesSettings` from `multiplatform-settings`
- **Fix:** Use `EncryptedSharedPreferences` from AndroidX Security for the token-specific preferences

- Alternative: Create a separate `EncryptedSharedPreferences` instance for tokens only, wrap in a `Settings`-compatible interface
- **Migration:** Existing plaintext tokens need to be migrated to encrypted storage on first launch

## Task 1: Fix foreground service null intent handling (B5)

**Action:** In `WorkoutForegroundService.kt`, handle the null intent case and consider switching to `START_NOT_STICKY`.

**Specific changes:**
1. Add null intent handling in `onStartCommand`:
   ```kotlin
   override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
       when (intent?.action) {
           ACTION_START_WORKOUT -> { /* existing start logic */ }
           ACTION_STOP_WORKOUT -> { /* existing stop logic */ }
           null -> {
               // System restarted service after kill — satisfy foreground contract then stop
               Logger.w { "WorkoutForegroundService restarted with null intent, stopping" }
               startForeground(NOTIFICATION_ID, createNotification("Workout service restarting..."))
               stopSelf()
           }
       }
       return START_NOT_STICKY  // Don't restart — killed workout can't meaningfully resume
   }
   ```
2. Changed to `START_NOT_STICKY` since a killed workout session cannot resume BLE connection or rep state
