<objective>
Verify the Ktor Darwin engine works for sync HTTP calls on iOS, confirm the TestFlight CI pipeline is compatible with the credential injection changes, and document any blockers for iOS deployment.
</objective>

<context>
- Phase context: `.planning/phases/30-ios-sync-launch/30-CONTEXT.md`
- Implementation plan: `docs/plans/mvp-cloud-sync-mobile.md` (Section 8)
- Branch: `MVP`
- Depends on: Plan 30-01 (credentials must be injected first)
- Ktor Darwin engine uses URLSession under the hood (different from OkHttp on Android)
- iOS CI workflows exist: `ios-testflight.yml`, `ios-testflight-internal.yml`, `ios-release-ipa.yml`
- Full iOS builds require macOS runner with Xcode
- TestFlight deployment requires Apple Developer credentials as GitHub secrets
- The shared Ktor client is configured in commonMain — engine selection is automatic per platform
</context>

<tasks>
### Task 1: Verify Ktor Darwin Engine Dependency

**Action**: Confirm the Ktor Darwin engine is properly declared in `shared/build.gradle.kts` for the iOS target.

Check for:
```kotlin
// In iosMain dependencies:
implementation(libs.ktor.client.darwin)
```

**What to verify**:
- The `ktor-client-darwin` dependency is declared in the iOS source set (not just commonMain)
- The version matches the other Ktor dependencies (3.3.3 per libs.versions.toml)
- No conflicting HTTP engine dependencies in the iOS target

**If missing**: Add the Darwin engine dependency to the iosMain source set.

**Verify**: Dependency declaration is present and version-consistent.

**Done when**: Ktor Darwin engine dependency is confirmed (or added) in the iOS source set.

---

### Task 2: Review iOS TestFlight CI Workflow

**File**: `.github/workflows/ios-testflight.yml`
**Action**: Read the workflow and verify it handles:

1. **Info.plist values**: The workflow copies Compose resources and builds the framework. Check if it modifies Info.plist during build (e.g., version bumping). Ensure our new SUPABASE_URL/SUPABASE_ANON_KEY entries aren't overwritten.

2. **Build steps**: Verify the workflow sequence:
   - `linkReleaseFrameworkIosArm64` — builds the shared KMP framework
   - Resource copying — Compose resources into framework bundle
   - XCFramework creation
   - Xcode archive + export
   - TestFlight upload

3. **Secrets required**: Document which GitHub secrets are needed (see mobile plan Section 8.3):
   - `BUILD_CERTIFICATE_BASE64`, `P12_PASSWORD`
   - `PROVISION_PROFILE_BASE64`, `KEYCHAIN_PASSWORD`
   - `TEAM_ID`, `PROVISIONING_PROFILE_NAME`
   - `APPSTORE_API_KEY`, `APPSTORE_API_KEY_ID`, `APPSTORE_ISSUER_ID`
   - `TESTFLIGHT_GROUP_NAME`

4. **Credential security**: Verify Info.plist credentials (publishable keys) don't need to be injected via CI secrets. Since they're publishable, embedding in Info.plist is acceptable.

**Verify**: Workflow is compatible with the Info.plist changes from Plan 30-01.

**Done when**: Workflow reviewed, compatibility confirmed, and any needed changes documented.

---

### Task 3: Document iOS Deployment Blockers

**Action**: Create a deployment readiness checklist documenting what's needed for a successful TestFlight submission:

1. **macOS requirement**: iOS builds cannot run on Windows CI runners
2. **Apple Developer credentials**: List which secrets are configured vs. missing
3. **Version alignment**: Info.plist shows `0.1.1` / `2025122504` — needs update to `0.7.0`
4. **Signing**: Distribution certificate + provisioning profile status
5. **Known risks**: Darwin engine untested for sync, Compose resource copying fragility

Write this as a section in the plan summary (not a separate file).

**Verify**: All blockers are documented with clear resolution paths.

**Done when**: Deployment blockers documented and actionable.
